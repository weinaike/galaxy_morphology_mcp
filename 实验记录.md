# 训练数据生成方案 — 对比实验记录

## 1. 实验目标

验证不同提议策略（rule_based / expert_guided / vlm_generated）及关键配置参数对训练数据生成质量的影响，为批量生产选定最优配置。

### 核心对比维度

| 对比 | 实验组 | 验证问题 |
|------|--------|---------|
| 提议策略价值 | E1 vs E3 | VLM提议相比规则提议能提升多少成功率？ |
| 专家规则对比 | E2 vs E3 | VLM提议能否匹配或超越基于Gadotti GT的专家规则？ |
| 专家提示注入 | E3 vs E4 | 在VLM prompt中注入Gadotti专家参数是否提升效果？ |
| Token成本优化 | E3 vs E5 | VLM并发调用从4路减到2路，成功率损失多少？ |
| 深层轨迹探索 | E3 vs E6 | 关闭贪心+放宽早停能否产出更多深层训练数据？ |

**E6 — vlm_generated + MH退火探索（深层轨迹）**
```python
PROPOSAL_STRATEGY = "vlm_generated"
USE_EXPERT_HINT_FOR_VLM = False
VLM_PROPOSAL_NUM_CALLS = 4
FORCE_GREEDY = False    # 关闭贪心，启用 Metropolis-Hastings 退火接受
MAX_PATIENCE = 2        # 保持默认，单变量对比
```

> **E6 设计动机**：E3 在 Step 4+ 成功率断崖归零，原因是严格贪心（`force_greedy=True`）不允许中间步暂时变差。
> 真实星系拟合中，加新component（如bar）常常导致chi2先升后降，需要多轮调参才能收敛。
> E6 仅改动一个变量（FORCE_GREEDY=False），与E3严格单变量对比：
>
> 1. **关闭贪心** (`FORCE_GREEDY=False`)：VLM判定 improvement=0 的变体仍有 ~13.5% 概率被接受
>    （Metropolis-Hastings: `P(accept) = exp(delta_r/T) = exp(-1/0.5) ≈ 0.135`），
>    允许pipeline走过"先变差再变好"的优化谷底，产出Step 4-6的训练样本。
>
> **E3 vs E6 关注的核心指标**：
> - 深层（Step 4-6）是否有 accepted 节点
> - 训练数据深度分布是否更均匀
> - SFT/DPO 样本数量变化（预期：E6 > E3）
> - 成本变化（预期：E6 ≈ E3，VLM调用数相同，仅搜索深度更深导致略多GALFIT运行）

**E7 — 模拟MCP交互式agent流程（单提议 + 全接受 + 完整历史）**
```python
EXPERIMENT_ID = "E7"
PROPOSAL_STRATEGY = "vlm_generated"
USE_EXPERT_HINT_FOR_VLM = False
VLM_PROPOSAL_NUM_CALLS = 1     # 一次只提议一个方案（像agent一步一步来）
ACCEPT_ALL = True               # 全部接受，不做reward门控
BEAM_TOP_K = 1                  # 单线程推进，不做beam搜索
VLM_HISTORY_MAX_STEPS = 0       # 注入全部历史（agent有完整上下文）

```

> **E7 设计动机**：E3-E6 的 VLM 提议了 add_bar（E6: 13次），但 0% 通过 reward。
> 原因是加 bar 后 chi2 先升后降，需要多步调参才能收敛，而我们的单步 reward 在第一步就拒绝了。
> MCP main 分支是交互式 agent 流程：agent 自己决策、自己判断、不被 reward 门卫拦截，
> 所以能走过"加bar→暂时变差→调参→恢复"的多步路径，拿到 70% 成分准确率。
>
> E7 模拟这个流程：NUM_CALLS=1（单提议，像agent一步步来）+ ACCEPT_ALL=True（不拦截）+
> 完整历史（VLM能看到"我刚加了bar，chi2变差了，下一步该调bar参数"）+ BEAM_TOP_K=1（单线程推进）。
> MAX_STEPS=15 允许走足够深来完成"加成分+调参"的多步过程。
>
> **E7 关注的核心指标**：
> - add_bar / add_bulge 是否能被**保留到最终模型**（而非第一步就被拒）
> - 成分准确率是否显著提升（对比 E6 的 ~20%）
> - 终点 chi2_nu / BIC 是否合理（全接受不代表终点差——VLM有完整历史，会自我修正）
> - 耗时与 token 消耗变化（NUM_CALLS=1 减少并发，但 MAX_STEPS=15 增加总步数）

> **E8 设计动机**：E6r 成分准确率仅 15%（3/20），核心瓶颈是 Bar recall=16%（19个有Bar的星系只检测到3个）。
> 根因分析：我们的 VLM 提议是**单轮调用**（4个分析阶段塞进1次 API 调用），Gemini 倾向跳过细致的
> 视觉分析直接输出"安全"的 modify/add_disk，几乎不提议 add_bar。
> 而 MCP main 分支的 `component_analysis` 是 **3轮对话**（Turn1:视觉特征提取→Turn2:参数审查+推理→Turn3:决策JSON），
> 同样用 Gemini 却能达到 70% 准确率。多轮对话**强迫** Gemini 在 Turn1 先做详尽的视觉观察（是否有"一字型/X型"bar残差），
> 然后在 Turn2 结合参数做推理，最后 Turn3 才输出决策。
>
> E8 vs E7 **单变量对比**：仅增加 `VLM_PROPOSAL_MULTITURN=True`，其余配置完全相同。
>
> **训练数据影响**：多轮模式下，每个 decision 额外携带 `thinking_chain` 字段（Turn1/Turn2/Turn3原文），
> 保存在 trajectory.json 的 `action_from_parent.thinking_chain` 中。提取 SFT 样本时，
> thinking_chain 作为 CoT 输出——训练时模型学到"先分析→再推理→输出JSON"的完整思维链，
> 但推理时仍是单步输出（model internalized CoT）。
>
> **API 成本**：每个 proposal 从 1 次 API 调用变为 3 次（Turn2/3不重复发图片，靠对话历史），
> prompt token 增约 50%，completion token 增约 2x。NUM_CALLS=1 时每步 3 次调用，可接受。
>
> **E8 关注的核心指标**：
> - add_bar 提议率是否显著提升（E6r 仅 2%，预期 E8 > 10%）
> - Bar recall 是否提升（E6r 仅 16%）
> - thinking_chain 质量：Turn1 视觉分析是否提及 bar 特征
> - SFT 样本的 thinking_chain 覆盖率

**E7r_rand — r 波段伪随机 20（与 E7r 对比：测选样红利）**
```python
EXPERIMENT_ID = "E7r_rand"
# 策略配置完全同 E7（全接受单提议单轮完整历史 MAX_STEPS=15）
TARGET_BANDS = ["SDSS_rband"]
GALAXY_ID_FILTER = []           # 种子42伪随机20（r波段train前20，非精选那20个）
```

> **E7r_rand 设计动机**：E7r 的 65% 是在 **MCP 精挑的 20 个 r 波段星系**上取得的，可能吃了**选样红利**
> （精选=好拟合代表）和 **prompt 对该批的隐性适配**。现有数据里"波段(g随机/r精选)"与"选样"完全耦合，
> 分不清 r 好是波段还是选样。
>
> 去耦手段：那 20 个精选星系无 g/i 波段版本，无法跨波段控制；改为**固定波段(r)、把选样从精选换成随机**。
> 借助 `get_train_test_split` 的**固定种子42 shuffle**（`dataset_utils.py:35-37`），`GALAXY_ID_FILTER=[]`
> 取到的 train 前20 就是**可复现的伪随机20**（非精选那批）。
>
> **对比结论读法**：E7r_rand ≈ 65% → 选样红利小、方法在 r 波段真行、全量 r 可期；明显掉 → 65% 吃了精选
> 红利，全量真实水平要下修。⚠️ 跑后需确认这 20 个与精选 20 的重叠度（伪随机+精选，期望重叠很小）。

### 5.3 抽取规则

**SFT**：遍历trajectory.json所有节点 → 跳过root → 跳过is_accepted=False → 跳过crash → 每个合格节点生成一条。

**DPO（树搜索内生，NUM_CALLS≥2 时）**：按parent_id分组 → winner池(is_accepted=True) × loser池(is_accepted=False且非crash) → 笛卡尔积配对 → 过滤confidence<0.7的loser。

**DPO（事后合成，NUM_CALLS=1 单链专用）**：单链无同父兄弟、树搜索内生 DPO=0，改用
`construct_dpo_from_sft.py` 事后构造。对每个 accepted 节点(=chosen)：从父节点重建输入状态 →
**弱模型**(flash)生成 rejected 候选 + **规则注入结构错误**(bar↔disk互换/删成分/加多余成分) →
每个候选实跑 GALFIT → **第一道客观筛**(BIC/chi2_nu 确认更差) → **第二道 vlm_reward(flash) 二次确认**
(剔除"其实改善了"的假负) → 成对输出 `all_dpo_synthetic.jsonl`。全量增量成本 ~$45(弱模型+GALFIT免费+flash确认)。
关键防坑：GALFIT 客观指标兜底避免假负样本毒化；结构错误负样本专治 bar 识别。

**终点代表叶子按 BIC 最低选**（不用"最深"）：BIC 内置过拟合惩罚，能避开"靠堆简并成分
走深"的过拟合死叶；BIC 缺失时回退 chi2_nu。没有任何 accepted 改进时回退到 root（初始拟合）。

**两层评估**（`--mode`）：
1. **客观指标层 (metric)**：纯比 BIC/chi2_nu，零成本秒出，无偏可复现，作为**主证据**。
   BIC 差异 >2.0 才算赢，否则看 chi2_nu，再否则平手。
2. **VLM 视觉层 (vlm)**：补 BIC 看不出的残差结构（旋臂/伴星系/偏心）。底层
   `reward.compare_two_fits_symmetric` 保留原 reward 全部天文知识，但对称化消偏
   （中性图片标签 Image A/B + 对称 prompt + 内部正反双向校验）。**双向矛盾不再判平手，
   如实标 `conflict` 并列出双向理由**，交客观指标定夺。

```bash
# 1) 先跑纯客观版（零成本秒出，拿无争议baseline）
python -m data_gen.evaluate_trajectory_compare \
  --exp-a output/E1__rule_based.../ \
  --exp-b output/E3__vlm_proposal.../ \
  --mode metric

# 2) 再跑完整版（客观+VLM双维度，默认 both）
python -m data_gen.evaluate_trajectory_compare \
  --exp-a output/E1__rule_based.../ \
  --exp-b output/E3__vlm_proposal.../ \
  --mode both --model gemini-3.1-pro-preview

# 输出 compare_E1_vs_E3.json:
#   metric_summary: 客观指标维度各胜场（主证据）
#   vlm_summary:    VLM维度各胜场 + conflicts(双向分歧)
#   details每星系含: metric_verdict/metric_basis, vlm_verdict, 双向理由, BIC/chi2/depth, 图路径
```

> 成本: metric 模式 0 token; both 模式每星系 2 次 VLM 调用（正反），20星系约 40 次。
> 结论读法: 以客观指标维度为准（如"E3 BIC胜13/20"），VLM 维度作旁证；两维度打架的
> 个别星系值得人工看图。
